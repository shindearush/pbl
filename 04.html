Mastering Control Flow: A Deep Dive into Python Custom Exceptions and Finally Blocks

By Code Daily

Writing code that works is easy; writing code that fails gracefully is an art. In Python, the `try-except` block is your primary defense against unexpected crashes. But what happens when the standard errors aren't specific enough for your logic?

In this comprehensive example, we define a custom exception called NegativeNumberError to enforce business logic. We then trigger a cascade of potential errors—ValueError, TypeError, and FileNotFoundError—to demonstrate how specific `except` blocks capture them. Finally, we use the finally block to ensure our cleanup code runs no matter what happens. 

# Defining a custom exception
class NegativeNumberError(Exception):
    pass

try:
    # ValueError example
    num = int(input("Enter a positive integer: "))
    if num < 0:
        raise NegativeNumberError("Negative numbers are not allowed")

    # TypeError example
    a = 10
    b = "5"
    c = a + b   # This will cause TypeError

    # FileNotFoundError example
    file = open("sample.txt", "r")
    file.close()

except ValueError:
    print("ValueError: Invalid input. Please enter an integer.")

except TypeError:
    print("TypeError: Cannot perform operation between different data types.")

except FileNotFoundError:
    print("FileNotFoundError: File does not exist.")

except NegativeNumberError as e:
    print("Custom Exception:", e)

except Exception as e:
    print("Unknown Error:", e)

else:
    print("Program executed successfully.")

finally:
    print("Execution completed.")

The beauty of this structure lies in the finally block. Even if the program crashes due to a TypeError on line 14, or raises our custom NegativeNumberError, the final print statement "Execution completed" will always run, ensuring resources are released.
